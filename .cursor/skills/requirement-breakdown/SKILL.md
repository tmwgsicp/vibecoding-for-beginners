# 需求拆解与 MVP 定义 Skill

> **状态:** 规划中 | 对应课程第 2、3 章  
> **作者:** 基于 AI News RSS 项目的需求拆解经验

---

## 核心原则

### 1. 识别核心假设
- 项目的核心假设是什么?
- 需要验证的最关键问题是什么?
- 什么是"有"就能用,"没有"就不成立的?

### 2. 拆解最小验证路径
- 验证假设最少需要哪些功能?
- 哪些功能是"锦上添花"?
- 哪些功能可以后续迭代?

### 3. 基于 3 周周期
- Week 1: 核心 MVP(能跑通)
- Week 2: 完善体验(能用得好)
- Week 3: 商业化准备(能赚钱)

---

## 工作流程

### Step 1: 接收需求描述

用户会描述他们想做的项目或功能。

### Step 2: 识别核心假设

通过提问帮助用户明确:
```
- 你的核心假设是什么?
- 用户的核心需求是什么?
- 如果只有一个功能,会是什么?
```

### Step 3: 拆解 MVP 功能

基于核心假设,列出:
1. **必须有的** (核心 MVP)
2. **最好有的** (体验优化)
3. **可以后续加的** (次要功能)

### Step 4: 转换成任务清单

```markdown
## MVP 任务清单 (Week 1)
- [ ] 核心功能 1
- [ ] 核心功能 2
- [ ] 核心功能 3
- [ ] 基础部署

## 体验优化 (Week 2)
- [ ] 错误处理
- [ ] 界面优化
- [ ] 性能优化

## 商业化 (Week 3)
- [ ] 用户系统
- [ ] 支付接入
- [ ] 监控和日志
```

## 真实案例模板

### AI News RSS 的拆解

**原始需求:**
"做一个 AI 资讯聚合工具"

**核心假设:**
用户需要自动筛选 AI 资讯的工具

**MVP 拆解:**

✅ **Week 1 MVP (必须)**
1. 爬取 AI 资讯源
2. 用 AI 筛选内容
3. 推送到用户邮箱

❌ **不是 MVP (后续)**
- 用户注册和登录
- 支付订阅
- 漂亮的界面
- 多语言支持

**验证结果:**
Week 1 上线 → 验证需求成立 → Week 2-3 补充商业化功能

## 常见场景处理

### 场景 1: 功能过多
```
用户: "我想做一个 AI 对话工具,支持:
- 多模型切换
- 联网搜索
- 图片生成
- 语音输入
- 对话记忆
- 知识库
- ..."

Skill 输出:
核心假设是什么?
→ 如果是"支持工具调用的对话工具"
→ MVP: 对话 + 一个工具(如联网搜索)
→ 其他功能后续迭代
```

### 场景 2: 需求模糊
```
用户: "我想做一个好用的工具"

Skill 引导:
- 好用指什么?(快?准确?界面好?)
- 解决什么问题?
- 目标用户是谁?
- 现有方案的不足是?
```

### 场景 3: 过度追求完美
```
用户: "MVP 要不要加用户系统?万一有人想注册呢?"

Skill 回应:
MVP 的目标是验证假设,不是做完整产品。
可以用这些方式绕过:
- 固定邮箱测试
- 邀请码内测
- 先做单用户版本
验证需求成立后,再加用户系统。
```
