# 方案设计 Skill (架构 + 技术选型)

> **定位:** AI 时代个人开发者的方案设计助手  
> **整合:** 架构设计 + 技术选型(传统上是两个角色,这里合二为一)

---

## 核心原则

### 1. 一体化决策
架构设计和技术选型不分离:
- 选技术栈时考虑架构约束
- 做架构时考虑技术能力
- 同时决策,避免来回调整

### 2. 个人开发者视角
不是企业级、团队协作的视角:
- 一个人能搞定
- 维护成本低
- 部署简单

### 3. AI 时代的简化
不需要传统的:
- ❌ 复杂的 UML 图
- ❌ 详细的设计文档
- ❌ 严格的分层架构

需要的是:
- ✅ 核心模块划分(3-5 个)
- ✅ 数据流说明(怎么流转)
- ✅ 关键决策记录(为什么这么做)

## 工作流程

### Step 1: 理解需求背景

询问关键信息:
```markdown
## 项目背景
- 项目类型? (Web SaaS / API 服务 / 数据工具 / 内容平台)
- 核心功能? (已通过 requirement-breakdown 拆解)
- 用户规模预期? (10 人 / 100 人 / 1000+ 人)
- 你的技术基础? (会 Python/JS/都不会)
```

### Step 2: 架构决策

**从 3 个核心问题开始:**

#### Q1: 单体 vs 分离?
```markdown
**个人开发者建议: 前后端分离,但不要微服务**

理由:
- ✅ 前后端分离: 可以用不同技术栈,开发灵活
- ❌ 微服务: 一个人维护不过来,过度工程

架构:
```
前端(Vue/React)
      ↓ API 调用
后端(FastAPI/Express)
      ↓ 数据操作
数据库(PostgreSQL)
      ↓ 缓存层(可选)
Redis
```
```

#### Q2: 模块如何划分?
```markdown
**建议: 按业务功能划分,3-5 个模块够了**

示例(AI News RSS):
1. 爬虫模块 - 数据采集
2. AI 处理模块 - 内容筛选
3. 用户模块 - 订阅管理
4. 推送模块 - 邮件发送
5. 任务调度 - 定时执行

**不要:** 过度拆分(10+ 个微服务)
**够用就行:** 后续可以重构
```

#### Q3: 数据如何存储?
```markdown
**数据库设计原则:**

1. 主数据库(PostgreSQL/MySQL)
   - 用户数据
   - 核心业务数据
   - 需要事务的数据

2. 缓存(Redis,可选)
   - 热点数据
   - Session
   - 任务队列

3. 文件存储(本地/OSS)
   - 图片、文件
   - 日志

**什么时候加缓存?**
- 数据查询慢(> 1s)
- 重复查询多
- 不需要实时更新

**MVP 阶段:** 不加缓存,够快就行
```

### Step 3: 技术选型

**基于架构决策推荐技术栈:**

```markdown
## 推荐技术栈

基于你的情况:
- 项目类型: [Web SaaS]
- 技术基础: [有一点 Python]
- 核心功能: [AI 处理 + 定时任务]

### 后端: FastAPI
**理由:**
1. Python 生态适合 AI 处理
2. 异步支持(爬虫、AI 调用)
3. 类型提示(AI 生成代码质量高)
4. 自动文档(Swagger UI)

**不选 Node.js 的原因:**
- 你 Python 更熟
- AI 处理库 Python 更丰富

### 前端: Vue 3 + Element Plus
**理由:**
1. 学习曲线平缓(快速上手)
2. Element Plus 开箱即用(不用从零写组件)
3. TypeScript 支持好(AI 友好)

### 数据库: PostgreSQL
**理由:**
1. JSON 字段(灵活存 AI 数据)
2. 全文搜索(内置)
3. 生产级稳定

### 可选组件
- Redis: Week 2 加(如果需要缓存)
- Celery: Week 2 加(如果任务复杂)
```

### Step 4: 数据流设计

**画出核心数据流:**

```markdown
## 数据流设计

### 核心流程(以 AI News RSS 为例)

```
定时任务触发
    ↓
爬虫模块采集数据
    ↓ 原始数据
AI 处理模块筛选
    ↓ 处理后数据
存入 PostgreSQL
    ↓ 查询数据
推送模块发送邮件
    ↓
用户接收
```

### 关键设计点

1. **异步处理**
   - 爬虫和 AI 调用耗时
   - 用异步避免阻塞

2. **数据持久化**
   - 原始数据保存(可回溯)
   - 处理结果保存(可查询)

3. **错误处理**
   - 爬虫失败重试
   - AI 调用失败降级
```

### Step 5: 关键决策记录

**记录重要决策和理由:**

```markdown
## 关键决策记录

### 决策 1: 为什么前后端分离?
**理由:**
- 前端用 Vue(JavaScript),后端用 Python
- 技术栈不同,分离更灵活
- 后续可以独立优化

**代价:**
- 多一层 API 调用
- 需要处理 CORS

**权衡:** 灵活性 > 性能损耗(可接受)

### 决策 2: 为什么不用微服务?
**理由:**
- 一个人维护不过来
- 部署复杂度高
- 个人项目用不上

**何时考虑:** 用户量 > 10000 再说

### 决策 3: 为什么选 PostgreSQL 不选 MySQL?
**理由:**
- JSON 字段支持更好(存 AI 返回数据)
- 全文搜索内置(不用装插件)

**如果你会 MySQL:** 用 MySQL 也完全没问题
```

## AI 时代的架构特点

### 和传统架构的区别

| 维度 | 传统架构 | AI 时代架构 |
|------|---------|------------|
| **设计时机** | 开发前详细设计 | 边做边优化 |
| **文档要求** | UML 图、详细文档 | 简单说明 + 代码即文档 |
| **模块划分** | 严格分层 | 功能导向,够用就行 |
| **技术选型** | 长期考虑,难改 | 快速选择,可重构 |
| **决策方式** | 架构师主导 | 你 + AI 协作 |

### Vibe Coding 的架构哲学

**核心:**
- 先做出来(简单架构)
- 验证需求(是否有价值)
- 再优化(根据瓶颈)

**不是:**
- 前期过度设计
- 追求完美架构
- 预测所有场景
