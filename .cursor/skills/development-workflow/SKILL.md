# 开发工作流 Skill (AI 时代版本)

> **定位:** 为个人开发者 + AI 协作设计的开发流程  
> **特点:** 快速迭代,而非传统瀑布模型

---

## 核心原则

### 1. 快速启动 > 完美准备
```
传统: 需求 → 设计 → 开发 → 测试 → 上线(每个阶段完整)
AI 时代: 需求 → 快速做 → 测试 → 上线 → 迭代(快速循环)
```

### 2. 边做边优化
- 不要前期过度设计
- 遇到问题再优化
- 用真实数据驱动改进

### 3. 你 + AI 的协作模式
- 你负责: 决策、判断、把控方向
- AI 负责: 实现、建议、技术细节

---

## 工作流程

### 🚀 Phase 1: 快速启动(Day 1-2)

#### Step 1.1: 明确要做什么
使用 `requirement-breakdown` Skill:
```
输入: 项目想法
输出: MVP 功能清单(3-5 个核心功能)
```

#### Step 1.2: 设计方案
使用 `solution-design` Skill:
```
输入: MVP 功能清单
输出: 架构设计 + 技术栈选择
```

#### Step 1.3: 搭建基础项目
**和 AI 协作:**
```
你: "基于刚才的方案,帮我搭建项目基础结构"
AI: [创建目录、初始化项目、配置文件]

检查清单:
- [ ] 前端项目初始化
- [ ] 后端项目初始化
- [ ] 数据库连接配置
- [ ] 第一个 Hello World
```

**用时:** 1-2 小时

---

### 💻 Phase 2: 核心 MVP(Day 3-7)

#### Step 2.1: 逐个功能实现

**不要一次做完,而是:**
```
Day 3: 功能 1(最核心)
    ↓ 测试可用
Day 4: 功能 2
    ↓ 测试可用
Day 5: 功能 3
    ↓ 集成测试
Day 6: 部署到服务器
Day 7: 真实环境测试
```

#### Step 2.2: 和 AI 的协作模式

**每个功能的开发流程:**
```
1. 你: 描述功能需求
2. AI: 建议实现方案(不是直接给代码)
3. 你: 理解方案,确认或调整
4. AI: 实现代码
5. 你: 测试和验证
6. 遇到问题: 使用 problem-diagnosis Skill
```

**关键:** 每个功能都要能跑通再做下一个

#### Step 2.3: MVP 检查清单

```markdown
- [ ] 核心功能全部能用
- [ ] 在本地环境测试通过
- [ ] 部署到服务器能访问
- [ ] 你自己能用一周(dogfooding)
- [ ] 至少 1 个真实用户测试

**如果都通过 → 验证需求 → 进入 Phase 3**
**如果不通过 → 修复问题 → 重新测试**
```

---

### ✨ Phase 3: 体验优化(Week 2)

#### Step 3.1: 收集真实反馈

**不要闭门造车:**
```
- 自己用一周,记录痛点
- 找 1-2 个朋友测试
- 记录他们的困惑和建议
```

#### Step 3.2: 优化方向

**基于反馈决定优化顺序:**

**优先级 1: 用户明确反馈的问题**
```
"这里看不懂"
"这个功能不好用"
"加载太慢"
```

**优先级 2: 你观察到的问题**
```
错误处理不友好
界面交互不顺畅
性能有瓶颈
```

**优先级 3: 锦上添花**
```
界面更好看
增加动画效果
多语言支持
```

#### Step 3.3: 技术优化

**遇到性能问题:**
- 使用 `performance-optimization` Skill
- 使用 `problem-diagnosis` Skill

---

### 💰 Phase 4: 商业化(Week 3,可选)

#### Step 4.1: 判断是否值得做

**关键指标:**
- 有没有人愿意付费?
- 使用频率如何?
- 用户反馈积极吗?

**如果都是"否" → 可能是伪需求,考虑 pivot**

#### Step 4.2: 补充商业化功能

```markdown
## 最小商业化功能

1. 用户系统
   - 注册/登录
   - 个人中心

2. 支付接入
   - 支付宝当面付(个人可用)
   - 订单记录

3. 基础监控
   - 错误日志
   - 用户行为统计
```

## AI 时代的开发特点

### 和传统开发的对比

| 维度 | 传统开发 | AI 协作开发 |
|------|---------|-------------|
| **需求阶段** | 详细 PRD 文档 | 简单描述 + AI 补充 |
| **设计阶段** | 详细设计文档 | 简单方案 + 边做边调 |
| **开发阶段** | 查文档写代码 | 描述需求,AI 实现 |
| **测试阶段** | 写测试用例 | 手动测试 + AI 生成测试 |
| **迭代速度** | 周/月级别 | 天级别 |

### 你不需要的传统流程

**❌ 不需要:**
- 详细的需求文档(PRD)
- 复杂的架构图(UML)
- 严格的代码评审
- 完整的测试覆盖

**✅ 需要:**
- 清晰的核心功能(3-5 个)
- 简单的架构说明(文字 + 简图)
- 能跑通的代码
- 关键路径测试

## 实战案例: AI News RSS 开发流程

### Week 1: MVP(Day 1-7)

**Day 1: 需求拆解**
```
核心功能:
1. 爬取 AI 资讯
2. AI 筛选
3. 邮件推送

技术方案:
- Python 爬虫
- 调用 AI API(Claude/MiniMax/DeepSeek 等)
- SMTP 邮件
```

**Day 2-3: 搭建基础 + 爬虫**
```
你: "用 Python 写一个爬虫,抓取 36kr 的 AI 资讯"
AI: [建议用 requests + BeautifulSoup]
你: "好,写代码"
AI: [实现爬虫]
你: 测试 → 能抓到数据 ✅
```

**Day 4: AI 处理**
```
你: "把抓到的数据发给 AI,让它筛选 AI 相关的"
AI: [实现 AI 调用,可用 Claude/MiniMax/DeepSeek]
你: 测试 → 能筛选 ✅
```

**Day 5: 邮件推送**
```
你: "把结果发邮件"
AI: [实现 SMTP]
你: 测试 → 能收到邮件 ✅
```

**Day 6-7: 部署**
```
你: "部署到服务器,每天早上 8 点自动执行"
AI: [Docker + cron]
你: 测试 → 能自动运行 ✅
```

**Week 1 结束:** MVP 上线,开始验证!

### Week 2: 优化

**基于使用反馈:**
- 邮件格式优化
- 增加订阅管理
- 添加错误处理

### Week 3: 商业化

**确认需求后:**
- 用户注册/登录
- 支付宝接入
- 订阅管理

## 和其他 Skills 的配合

### 完整开发流程

```
开始项目
    ↓
requirement-breakdown      # 做什么?
    ↓
solution-design            # 怎么做?用什么?
    ↓
development-workflow       # 跟着流程做(当前 Skill)
    ↓ 遇到问题
problem-diagnosis          # 出错了怎么办?
    ↓ 需要优化
performance-optimization   # 性能优化(全栈)
    ↓
完成迭代!
```

### 何时用哪个 Skill?

**项目启动阶段:**
- requirement-breakdown(必用)
- solution-design(必用)
- development-workflow(可选,新手用)

**开发过程中:**
- problem-diagnosis(按需)
- performance-optimization(按需)

**经验丰富后:**
- 可以跳过 development-workflow
- 直接和 AI 对话
- 遇到卡点才用 Skills

## 给小白的引导

### 如果你是第一次做项目

**跟着这个流程:**

```markdown
## Day 1: 规划(2-3 小时)
1. 用 requirement-breakdown 拆解需求
2. 用 solution-design 设计方案
3. 列出 Week 1 的任务清单

## Day 2-7: 实现 MVP(每天 2-4 小时)
每天一个功能:
- 早上: 和 AI 讨论实现方案
- 上午: AI 帮你实现
- 下午: 测试和调试
- 晚上: 能跑通就进入下一个

## Week 2: 优化(弹性时间)
- 自己用一周
- 根据体验优化
- 找朋友测试

## Week 3: 上线(如果要做)
- 补充商业化功能
- 正式发布
```

- 跳过 Skills,直接对话

## 常见问题

### Q: 一定要按这个流程吗?

A: **不一定!**

这个流程是给小白的"抓手":
- 不知道从哪开始 → 跟着流程走
- 有经验了 → 按自己的节奏

**记住:** Vibe Coding 强调灵活性,不是死板流程

### Q: 我想跳过某个步骤可以吗?

A: **完全可以!**

比如:
- 需求很简单 → 跳过需求拆解
- 技术栈确定 → 跳过技术选型
- 很小的项目 → 不需要架构设计

**用你的判断,Skills 只是建议**

### Q: 和传统敏捷开发的区别?

A:

| 维度 | 敏捷开发 | AI 协作开发 |
|------|---------|-------------|
| 迭代周期 | 2 周一个 Sprint | 1 天一个功能 |
| 文档要求 | User Story + 验收 | 简单描述即可 |
| 代码评审 | 必需 | AI 实时审查 |
| 测试 | TDD/自动化测试 | 手动 + AI 生成 |

**更快,更灵活,更适合个人开发者**
